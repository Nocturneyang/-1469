# 多平台社媒监控系统 — 结构化 7 步开发方案

> **文档说明**：本方案深入整合了《社媒监控可行性分析》与现存系统架构设计成果。以结构化的 7 阶段工作流（Discovery → Codebase Exploration → Clarifying Questions → Architecture Design → Implementation → Quality Review → Summary）作为新系统落地的**最高执行与开发蓝图**。

---

## Phase 1: Discovery — 核心目标与破局思路

### 1. 业务愿景
将只支持 WhatsApp 的单一监控工具，升级为覆盖 WhatsApp / Telegram / 企微 / Teams 的**全平台群聊通讯智能监控枢纽**。依据「两类群体独立分析」的业务场景划分：
1. **供应商群（Supplier）**：快速预警路由异常、CR(转化率) 跌零、通道调整等**秒级/分钟级技术事故**。（告警目标：内部即时告警）
2. **客户群（Client）**：精准捕捉客户诉求、投诉、需求询价，并向 CRM 沉淀**长效客户画像**。（告警目标：内部告警 + CRM 工单/画像标签）

### 2. 破局思路：从“采集即分析”到“三层解耦”
旧版系统将“接收消息 → AI 分析 → 推送”紧密绑定。一旦 AI API 卡顿，或判断提示词（Prompt）出错，消息便会永久漏采或误判。
**新系统确立了最高优先级原则：全量采集 → 持久存储 → 后置分析。**
- **采集层职责坚守**：所有平台均以群聊消息监控为主，坚决不漏消息，保留原始格式，连换行和附件元数据均不能抛弃。
- **存储即基石**：存储原汁原味的聊天数据资产，是日后能够四种模式灵活应用的前提（**实时窗口、定时批量、手动触发、系统回溯**）。

## Phase 2: Codebase Exploration — 站在巨人的肩膀上（资产盘点）

目前旧版 `whatsapp-monitor v2.3.1` 核心组件（主要是 index.js）的改造策略已经确立：

| **资产类型** | **复用决策** | **提取行动** |
| --- | --- | --- |
| **💡 AI 提示词系统** | **重构复用 (40%)** | 现有模板成熟，将基于此拆分出**两套独立引擎**和对应的 Prompt（供应商专属技术指标模板 vs 客户专属需求分析模板）。 |
| **📢 钉钉/企微 推送** | **直接复用 (100%)** | 钉钉 Markdown 告警模板、Webhook 鉴权逻辑完全成熟，直接提取至独立 `alert/` 模块。 |
| **⚡ 优先级配置管理** | **直接复用 (100%)** | 现存的优先级合并逻辑继续沿用。 |
| **🔴 纯 JSON 数据存储** | **果断废弃 (0%)** | 原 `records.json` 极易引发并发写入瓶颈及数据丢失，这违背全量持久化初衷，必须全盘替换为并启用 WAL 模式的 SQLite。 |
| **⚙️ 消息死循环耦合** | **果断废弃 (0%)** | 将原 `message_create` 中触发 AI 和告警的代码链条彻底斩断。 |

## Phase 3: Clarifying Questions — 启动前必须落实的前提

在正式编写新架构代码前，由于多平台的异构性，需立刻扫清环境与基础设施障碍：

1. **底层物理资源保障**：1 台至少 2C4G、海外节点、具有**公网 IP** 的云服务器（企微和 Teams 回调强依赖公网）。
2. **各平台权限通道打通**：
   - **Telegram**：完成 @BotFather 创建，并关闭隐私模式（Privacy Mode disabled）以便 Bot 收取所有群消息，配置极为简单无需申请。
   - **企微 (WeCom)**：确认企业后台管理员协助创建自建应用、配置公网 URL 回调并获取解密密钥(EncodingAESKey)，确认能否购买或试用“会话内容存档”增值服务。
   - **Teams**：确认具备 M365 订阅，完成 Azure AD 应用注册，安装 Bot 使用基于 RSC (Resource-Specific Consent) 模型的 `ChannelMessage.Read.Group` 权限，无需最高级的 Tenant-wide 许可。
3. **数据沉淀对接**：明确目标 CRM 系统的 API 对接规范文档（如客户 ID 清单分类收集规则、标签写入格式及工单接口规范）。
4. **历史包袱清算**：基于跨度较大的基建级改造，系统建议旧版老旧数据彻底放弃兼容：**新老划断，空库开局**。

## Phase 4: Architecture Design — 最终架构蓝图

通过参考各平台的可行性调研与能力矩阵反馈，确立如下核心架构方案。

### 1. 核心稳定性设计：PM2 进程级隔离护航（Process Isolation）
由于 WhatsApp (`whatsapp-web.js`) 基于 Puppeteer + Chromium Web 逆向，极吃内存且易引发 OOM 隐性崩溃。为了防止“火烧连营”（WA 崩溃将高质量的稳定官方 API 服务也波及中断），明确采用 **进程级物理隔离** 编排：
- 主控进程：`server.js` (掌控分析分发与 Web 面板监控)
- A 类“高危/易崩”进程：`worker-wa.js` (通过机制强制隔离，设定内存阀值自动回收 `max_memory_restart: '1G'`)
- B 类“轻量/稳定”进程：`worker-tg.js`, `worker-wecom.js`, `worker-teams.js`

### 2. 数据库设计抽象：核心骨架引擎
实施 SQLite `WAL` 模型保证高并发写入响应处理机制。数据库划分为四大领域核心模型层：
- `messages`【全局基座】：底层全量原始表保存不可磨灭的记录集，依赖双唯一键(`platform, message_id`) 防重死循环。
- `analysis_results`：独立表分离承载两类 AI 引擎产出结论（实时级、天级及周批处理分析报表），并附带对应回溯版本的 Prompt_version。
- `profiles` 联合 `pattern_reports` 层：通过大数据聚类产生画像：包括对供应商体系的服务体检报告，并产生客户侧情绪指数体系输出。
- `group_partner_map`（群/实体映射）：作为多平台打通内部系统的唯一映射表（将分散在各个即时通讯的标识如 Group_ID，收束为一个关联公司内体系的内部 Partner/CRM_ID）。

### 3. 三层立体维稳健康屏障 (Health Defense Network)
防止“幽灵假死”（进程健在，收发停工）全面设计系统安控网雷达：
- **物理容能灾备保障**：使用底层系统指令调度恢复并自动隔离错误组件内存漏洞。
- **软件侧心跳侦探**：开发专用 `heartbeat.js` 与平台无声监测心跳（若高频群组离奇安静长达异常临界时间则爆点掉线故障进行维护预警）。
- **源头依赖探针**：挂载 `github-watcher.js` 常态爬取监控依赖项官方安全与新升级风向标（如 `whatsapp-web.js` 更新或企微 API 改版），实现未雨绸缪主动排雷防护体系。

## Phase 5: Implementation — 十步重构落地法

我们将此系统构建的长跑实操转化为按部就班的十条明亮主线：

1. **总骨架搭建系统生态圈**：布置 `social-monitor/` 生态目录结构及全局指挥中心 PM2 `ecosystem.config.js` 。
2. **数据底座稳固确立**：部署全系列 SQLite 初始化程序（强调启用防锁队列模式 WAL），部署四大维表矩阵框架体系。
3. **组件基石接口适配器构建**：创建共用核心协议 `adapters/base.js`规范抽象全生命周期，统一定制基础级报警分发引擎 `alert/alerter.js`。
4. **【首个里程碑：采集集群试跑打样】**：针对旧代码重新剥离包装产生独立的 WA Worker 及超低难度的 TG Worker 进行双平台原始级对接联调（**唯一执行：标准入库**）。
5. **解开旧羁绊死脉门**：将老版监控触发的所有即时处理判定环节（如即时 AI 生成反馈触发链等高并发漏洞）全部连根物理抹除，贯彻单行“只存不问”规则基准。
6. **双芯智能分拆化运作体系搭建**：主调配任务分流接管原始库未读资料到两大智库 `supplier-analyzer.js`与 `client-analyzer.js` 开展无害并行后置 AI 分析。
7. **数据视窗观测塔建立**：部署前后分离 Web 可化界面直观呈递「供应商渠道效能总览屏」及「核心大客户舆情探测台」。
8. **跨界联动 CRM 后置触角开发**：组编规律性自动探测脚手架抽提跨系同源指标（画像/体检大盘）建立 CRM `crm.js` 推送桥接端口。
9. **强权限商业端涉水与攻坚战**：专人跟踪推进合规许可：开发接轨具备企业加密级别限制通讯环境要求架构设计开发相应的 企微 Worker 应用及 Teams  采集接入节点配置验证。
10. **体系排毒及精准调速回演调试**：激活全部策略池的回溯纠偏执行预案并展开真库试冲验证来检验消除高危重跑时所暴漏的所有细小疏忽性 Bug 。

## Phase 6: Quality Review — 防御性编程体系检查

本着防患于未然方针在编写期间严查以下系统侧漏死角红线：

- ☑️ **离线重连包涌入雪崩屏障**：由于弱网络造成的频繁复连而激发的集中下放多台重组陈年旧短讯进行重复的“攻击式录入处理”，强卡物理索引唯一主键。
- ☑️ **被埋没的重要线索媒体追踪**：系统接收无法解析内容的音频及多媒体图像决不可抛弃或短路拦截，全部原模原块进行日志化索引登记到对应附件媒体中心以便后期反向人工索证查找与分析支撑点依据。
- ☑️ **短促惊慌口令的特级通行过滤策略**：杜绝“过滤短于 X 个单词就是无意义无价值”等暴力排噪逻辑思维（业务核心中最常产生关键爆雷情况仅是一句单纯的 ‘0’ 或直接发 ‘down’等简短暗喻），交给最后道门槛 AI 大数据智能洗炼研磨出真金降噪而不是进行基础门槛直接隔离销毁。
- ☑️ **防止连锁阻断 AI 黑洞死循环防范网**：设立调用模型外网超时等各类异常网络请求 API 断供状况识别监测器抓截如 HTTP `429 Too Many Requests` 以及 `402 Payment Required` 模型资源耗尽故障引发不断挂机无限死机耗空情况（进行紧急短缩系统预案与快速止损失联报告推送）。

## Phase 7: Summary & Milestone Roadmap — 执行时间线

全流程以阶梯状排期总计长跑覆盖周期预期为 `7-8 周`。策略为核心以数据流的「通达」至「精准」：

### 阶段 1 (P1)：数据命脉铺底与老旧核心置换 (历时约 2.5 周)
**【核心交付物】**：数据流接收器和判读引擎安全平滑斩断切割；通过多进程隔离调度统筹建立完成 `WA + TG` 系统独立收信上云体系。配属初显规模可视数据库操控观测底座机制模型基础（确立全库数据的正常收集存续健康环境）。

### 阶段 2 (P2)：双高端企业通道接入合规战 (历时约 1.5 周)
**【核心交付物】**：顺利推进行政许可并克服部署技术关卡堡垒，搭建企微内网解密转换协议及海外 Microsoft Teams 的 RSC 数据接入桥梁获取所有正规受控信息流体系完成四大网通汇集成源池计划。

### 阶段 3 (P2.5 & P3)：智能矩阵铺设及联运互通 (历时约 3 周)
**【核心交付物】**：
- P2.5：核心引擎开启常态业务工作组双边并发工作制（专门设立的供应与客户不同判定准线执行自动巡查纠误）。
- P3：积累充分历史基数开展后方定期规律汇总：按客户按需组别出具画像报表反馈并顺利连入后端 CRM 信息中心。

### 阶段 4 (P4)：柔性优化迭代精馏冲刺 (历时约 1 周)
**【核心交付物】**：彻底显现「数据不灭优势池」。基于海量存库历史案例资源采用无风险沙盘模拟技术反向检验喂测并升级进化多种智能判断 Prompt 标准公式模型体系；结合系统试压不断提高报警捕获灵敏度的绝对响应并压低噪声，最终交出全覆盖，高精准的全位监控雷达中控塔防体系大成。
